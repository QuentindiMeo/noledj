# M3 — DDL et contraintes

## Objectif

À la fin de ce module, l'apprenant sera capable de :

- Distinguer **DDL** (Data Definition Language) et **DML** (Data Manipulation Language).
- Créer une table avec **`CREATE TABLE`** et tous les types de contraintes.
- Modifier la structure avec **`ALTER TABLE`** (ajout/suppression/modification de colonnes).
- Supprimer une table avec **`DROP TABLE`** en comprenant les implications.
- Maîtriser les **5 types de contraintes** : `PRIMARY KEY`, `FOREIGN KEY`, `UNIQUE`, `NOT NULL`, `CHECK`.
- Appliquer des **conventions de nommage** cohérentes sur tables, colonnes, contraintes, index.

## Durée estimée

0,75 jour.

## Pré-requis

- M1 et M2 SQL terminés.

## 1. DDL vs DML

### Distinction

| Catégorie | Sigle | Objet | Exemples |
|-----------|-------|-------|----------|
| **DDL** | Data Definition Language | Structure de la base | `CREATE`, `ALTER`, `DROP`, `TRUNCATE` |
| **DML** | Data Manipulation Language | Contenu des tables | `SELECT`, `INSERT`, `UPDATE`, `DELETE` |
| **DCL** | Data Control Language | Permissions | `GRANT`, `REVOKE` |
| **TCL** | Transaction Control Language | Transactions | `BEGIN`, `COMMIT`, `ROLLBACK` |

**M2** a couvert le DML. **M3** se concentre sur le DDL — comment **construire** la base elle-même.

**Analogie.** Le DDL pose les murs et la plomberie de la maison. Le DML emménage les meubles et fait à manger. On ne peut pas inverser : sans murs, pas de cuisine.

### Caractéristique importante

En PostgreSQL et la plupart des SGBD, le DDL est **transactionnel** : on peut faire un `CREATE TABLE` puis un `ROLLBACK` et la table disparaît. Très pratique pour les migrations risquées.

MySQL en revanche fait un **commit implicite** avant et après tout DDL — pas de rollback possible. À surveiller selon le SGBD.

## 2. `CREATE TABLE` — créer une table

### Syntaxe minimale

```sql
CREATE TABLE users (
    id        INTEGER PRIMARY KEY,
    email     VARCHAR(255) NOT NULL,
    name      VARCHAR(100) NOT NULL
);
```

### Syntaxe étendue

```sql
CREATE TABLE users (
    id          SERIAL PRIMARY KEY,
    email       VARCHAR(255) UNIQUE NOT NULL,
    name        VARCHAR(100) NOT NULL,
    age         INTEGER CHECK (age >= 0 AND age <= 150),
    role        VARCHAR(20) NOT NULL DEFAULT 'user',
    is_active   BOOLEAN NOT NULL DEFAULT true,
    created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
    updated_at  TIMESTAMP
);
```

Chaque ligne : `<nom_colonne> <type> [contraintes]`.

### `CREATE TABLE IF NOT EXISTS`

```sql
CREATE TABLE IF NOT EXISTS users ( ... );
```

Ne lève pas d'erreur si la table existe déjà. Utile dans les scripts de migration idempotents.

### Auto-incrément selon SGBD

| SGBD | Syntaxe |
|------|---------|
| PostgreSQL | `SERIAL` (alias de `INTEGER + DEFAULT nextval(seq)`), ou `BIGSERIAL`, ou `GENERATED BY DEFAULT AS IDENTITY` (standard SQL) |
| MySQL | `INTEGER AUTO_INCREMENT` |
| SQLite | `INTEGER PRIMARY KEY AUTOINCREMENT` (note : `INTEGER PRIMARY KEY` suffit souvent) |
| SQL Server | `INT IDENTITY(1,1)` |

La syntaxe standard moderne `GENERATED BY DEFAULT AS IDENTITY` est supportée par PostgreSQL ≥ 10 et SQL Server.

## 3. Les 5 types de contraintes

### 1. `NOT NULL`

La colonne **doit** avoir une valeur. Toute tentative d'`INSERT` ou `UPDATE` qui laisse `NULL` est rejetée.

```sql
name VARCHAR(100) NOT NULL
```

Règle de bonne hygiène : par défaut, **toute colonne devrait être `NOT NULL`** sauf si l'absence de valeur a un sens métier explicite.

### 2. `UNIQUE`

Toutes les valeurs de la colonne doivent être uniques (avec la subtilité que `NULL` est considéré unique par défaut — plusieurs `NULL` sont autorisés).

```sql
email VARCHAR(255) UNIQUE
```

**Unique composite** (sur plusieurs colonnes) :

```sql
CREATE TABLE memberships (
    user_id INTEGER,
    team_id INTEGER,
    role    VARCHAR(20),
    UNIQUE (user_id, team_id)        -- pas deux fois le même (user, team)
);
```

### 3. `CHECK`

Une expression booléenne qui doit être vraie. Le SGBD rejette toute insertion qui la viole.

```sql
age INTEGER CHECK (age >= 0 AND age <= 150)
```

**CHECK sur plusieurs colonnes** :

```sql
CREATE TABLE products (
    price       DECIMAL(10, 2),
    discounted  DECIMAL(10, 2),
    CHECK (discounted <= price)
);
```

### 4. `DEFAULT`

Valeur par défaut si la colonne n'est pas fournie à l'INSERT.

```sql
created_at  TIMESTAMP NOT NULL DEFAULT NOW(),
role        VARCHAR(20) NOT NULL DEFAULT 'user'
```

À combiner systématiquement avec `NOT NULL` pour les timestamps et drapeaux.

### 5. `PRIMARY KEY` et `FOREIGN KEY`

Plus complexes — sections 4 et 5.

## 4. Clé primaire (`PRIMARY KEY`)

### Théorie

La **clé primaire** est l'**identité unique** d'une ligne dans une table. Elle est :

- **Unique** (jamais deux fois la même valeur).
- **Non NULL** (toujours renseignée).
- **Indexée automatiquement** (recherche rapide).
- **Unique par table** (une seule PK par table).

**Analogie.** Le numéro de sécurité sociale. Identifiant stable, unique, qui ne change jamais — même si tu changes de nom ou d'adresse.

### Déclaration

**Sur une seule colonne** :

```sql
CREATE TABLE users (
    id    SERIAL PRIMARY KEY,
    email VARCHAR(255) UNIQUE NOT NULL
);
```

**Composite** (sur plusieurs colonnes) :

```sql
CREATE TABLE memberships (
    user_id  INTEGER,
    team_id  INTEGER,
    role     VARCHAR(20),
    PRIMARY KEY (user_id, team_id)
);
```

Une `MEMBERSHIP` est identifiée par le couple (user, team). Pas deux fois la même paire possible.

### Surrogate key vs natural key

- **Natural key** : une colonne qui a un sens métier (ISBN d'un livre, email d'un user).
- **Surrogate key** : un id artificiel généré (auto-incrément, UUID).

**Pratique moderne** : surrogate key (`id SERIAL`) **toujours**, plus une `UNIQUE` sur la colonne métier (`email`, `isbn`) si applicable.

Pourquoi : les natural keys peuvent **changer** (un user change d'email, un ISBN est corrigé). Une surrogate key est immutable par nature.

### UUID vs integer

```sql
id UUID PRIMARY KEY DEFAULT gen_random_uuid()    -- PostgreSQL
id INTEGER PRIMARY KEY AUTOINCREMENT             -- SQLite
```

| Aspect | Integer auto-incrément | UUID |
|--------|------------------------|------|
| Lisibilité | Facile (`123`) | Difficile (`a3f...`) |
| Performance index | Excellente | Moins bonne |
| Génération distribuée | Non (besoin de coordination) | Oui (chaque service génère) |
| Devinabilité | Facile (`/users/124`) | Impossible |
| Taille | 4 ou 8 octets | 16 octets |

**Heuristique** :

- Application monolithique → `SERIAL` / `BIGSERIAL`.
- Système distribué multi-services → `UUID`.
- API publique où la non-devinabilité est importante → `UUID`.

## 5. Clé étrangère (`FOREIGN KEY`)

### Théorie

Une **clé étrangère** est une colonne (ou groupe de colonnes) qui **référence** la clé primaire d'une autre table.

```sql
CREATE TABLE orders (
    id        SERIAL PRIMARY KEY,
    user_id   INTEGER NOT NULL REFERENCES users(id),
    total     DECIMAL(10, 2) NOT NULL
);
```

Le SGBD **garantit** que toute valeur de `orders.user_id` correspond à un `users.id` existant. Tentative d'insérer un `user_id` inexistant → erreur.

**Analogie.** Une référence à un autre document. Si on cite "voir lettre n°42", la lettre 42 doit exister. Sans cette garantie (intégrité référentielle), on aurait des références mortes partout.

### `ON DELETE` et `ON UPDATE`

Que faire quand la ligne **référencée** (parent) est supprimée ou modifiée ?

```sql
user_id INTEGER NOT NULL
    REFERENCES users(id)
    ON DELETE CASCADE
    ON UPDATE CASCADE
```

| Action | Effet quand le parent est supprimé/modifié |
|--------|---------------------------------------------|
| `CASCADE` | Supprime / met à jour les lignes enfants automatiquement |
| `RESTRICT` ou `NO ACTION` | Refuse la suppression / modification du parent (défaut) |
| `SET NULL` | Met `NULL` sur la colonne enfant |
| `SET DEFAULT` | Met la valeur DEFAULT sur la colonne enfant |

### Cas typique

```sql
CREATE TABLE order_items (
    id        SERIAL PRIMARY KEY,
    order_id  INTEGER NOT NULL
        REFERENCES orders(id) ON DELETE CASCADE,
    product_id INTEGER NOT NULL
        REFERENCES products(id) ON DELETE RESTRICT,
    quantity  INTEGER NOT NULL CHECK (quantity > 0)
);
```

- `ON DELETE CASCADE` sur `order_id` : supprimer une `order` supprime ses `order_items`. Cohérent (les lignes n'existent pas sans la commande).
- `ON DELETE RESTRICT` sur `product_id` : on ne peut pas supprimer un produit s'il est encore référencé. Protection métier.

### FK composite

Si la table référencée a une PK composite, la FK doit l'être aussi :

```sql
CREATE TABLE membership_logs (
    user_id  INTEGER,
    team_id  INTEGER,
    action   VARCHAR(50),
    FOREIGN KEY (user_id, team_id) REFERENCES memberships(user_id, team_id)
);
```

### Anti-pattern à éviter

Beaucoup de bases legacy **n'ont pas de FK** (par crainte de la performance, ou par négligence). Conséquence : intégrité référentielle non garantie au niveau SGBD → bugs métier silencieux. **Toujours déclarer les FK** ; les enlever par optimisation est une décision rare et documentée.

## 6. `ALTER TABLE` — faire évoluer la structure

### Ajouter une colonne

```sql
ALTER TABLE users
ADD COLUMN phone VARCHAR(20);
```

Avec contrainte :

```sql
ALTER TABLE users
ADD COLUMN phone VARCHAR(20) NOT NULL DEFAULT '';
```

Note : ajouter une colonne `NOT NULL` **sans** `DEFAULT` sur une table non vide **échoue** — il faut soit fournir un default, soit faire en deux étapes (add nullable, backfill, alter to not null).

### Supprimer une colonne

```sql
ALTER TABLE users
DROP COLUMN phone;
```

**Irréversible** sans backup. À utiliser après vérification stricte.

### Renommer une colonne

```sql
ALTER TABLE users
RENAME COLUMN name TO full_name;
```

### Changer le type

```sql
ALTER TABLE users
ALTER COLUMN age TYPE BIGINT;
```

Avec conversion explicite si nécessaire :

```sql
ALTER TABLE users
ALTER COLUMN age TYPE TEXT USING age::TEXT;
```

### Ajouter / supprimer une contrainte

```sql
ALTER TABLE users
ADD CONSTRAINT users_email_unique UNIQUE (email);

ALTER TABLE users
DROP CONSTRAINT users_email_unique;

ALTER TABLE users
ADD CONSTRAINT users_age_check CHECK (age >= 0);
```

### Migrations en production

Les `ALTER TABLE` lourdes (`ADD COLUMN NOT NULL` sur table de 1 milliard de lignes) peuvent **bloquer la table** pendant la durée de l'opération — minutes ou heures. À gérer via :

1. Outils de migration spécialisés (Flyway, Liquibase, Alembic).
2. Migrations en plusieurs étapes (`add nullable` → `backfill` → `alter to not null`).
3. Stratégies de migration online (pt-online-schema-change pour MySQL).

Sujet approfondi au niveau Senior.

## 7. `DROP TABLE` — supprimer une table

### Syntaxe

```sql
DROP TABLE users;
```

**Irréversible**. Toutes les données disparaissent.

### Avec contrôle

```sql
DROP TABLE IF EXISTS users;
```

Ne lève pas d'erreur si la table n'existe pas. Idéal pour les scripts de rebuild.

### Avec dépendances

```sql
DROP TABLE users CASCADE;
```

Supprime aussi tous les objets qui dépendent de `users` (vues, FK pointant vers elle). À utiliser avec extrême prudence — on peut casser beaucoup de choses d'un coup.

```sql
DROP TABLE users RESTRICT;  -- défaut : refuse si quelque chose dépend de la table
```

### Garde-fous

- Toujours faire un `\dt` ou `SHOW TABLES` avant pour vérifier qu'on cible la bonne base / le bon schéma.
- En production : `BEGIN; DROP TABLE users; -- vérifier ; ROLLBACK; -- ou COMMIT;`.
- Toujours **un backup** récent disponible.

## 8. Conventions de nommage

### Pourquoi des conventions

Un schéma sans conventions devient illisible passé 20 tables. Chacun nomme à sa façon (`User`, `users`, `usr`, `tbl_user`...). Pour collaborer, il faut une **règle unique**.

### Convention courante (PostgreSQL et la plupart des projets modernes)

| Objet | Convention | Exemple |
|-------|------------|---------|
| **Table** | `snake_case`, pluriel | `users`, `order_items` |
| **Colonne** | `snake_case`, singulier | `user_id`, `created_at` |
| **PK** | `id` (la table parente est implicite) | `users.id` |
| **FK** | `<table_référencée_singulier>_id` | `user_id` (référence `users.id`) |
| **Index** | `idx_<table>_<colonnes>` | `idx_orders_user_id` |
| **Contraintes** | `<table>_<colonnes>_<type>` | `users_email_unique`, `orders_total_check` |
| **Trigger** | `tg_<table>_<event>` | `tg_users_updated_at` |
| **Vue** | `vw_<nom>` ou simplement `<nom>` | `vw_active_users` |

### Snake_case vs camelCase

PostgreSQL convertit les identifiants en lowercase par défaut. Donc `userId` devient `userid` à l'exécution, ce qui pose des problèmes de lisibilité. **`snake_case` est l'idiome universel SQL**.

Si on **doit** utiliser camelCase (rare, projet legacy), il faut entourer de doubles guillemets :

```sql
CREATE TABLE "Users" (
    "userId" SERIAL PRIMARY KEY,
    "userName" VARCHAR(100)
);

SELECT "userName" FROM "Users";
```

Très pénible à long terme. À éviter.

### Singulier vs pluriel

Débat religieux. Deux écoles :

- **Pluriel** (`users`, `orders`) — majoritaire, suggère "la table contient plusieurs users".
- **Singulier** (`user`, `order`) — argument "ORM-friendly", la table décrit un user.

Tant qu'on est **cohérent** dans un projet, peu importe le choix. Le pluriel est plus répandu en 2025.

### Mots réservés

Éviter les noms qui collisionnent avec des mots-clés SQL : `order`, `user`, `group`, `type`, `select`. Préférer `orders`, `users`, `groups`, `categories`, etc.

## 9. Exercices pratiques

### Exercice 1 — Créer une table simple (≈ 20 min)

Créer la table `products` :

- `id` auto-incrément, clé primaire.
- `sku` (string 20 chars), unique, obligatoire.
- `name` (string 200), obligatoire.
- `price` (decimal 10,2), obligatoire, > 0.
- `category` (string 50), valeur par défaut `'general'`.
- `created_at` timestamp, défaut `NOW()`.
- `updated_at` timestamp, nullable.

Insérer 3 produits. Vérifier que :

- L'insertion d'un produit sans `name` est rejetée.
- L'insertion avec un `sku` dupliqué est rejetée.
- L'insertion avec `price = -5` est rejetée.

### Exercice 2 — Foreign key et CASCADE (≈ 30 min)

Créer 2 tables :

- `categories` (id, name unique).
- `products` (id, name, price, category_id FK vers categories).

1. Choisir `ON DELETE CASCADE` ou `RESTRICT` pour la FK et justifier.
2. Insérer 2 catégories et 5 produits.
3. Tenter de supprimer une catégorie qui a des produits — observer le comportement.

### Exercice 3 — Évolution de schéma (≈ 30 min)

Sur une table `users` existante (de M2) :

1. Ajouter une colonne `phone VARCHAR(20)` nullable.
2. Ajouter une colonne `country VARCHAR(2) NOT NULL DEFAULT 'FR'`.
3. Renommer `name` en `full_name`.
4. Ajouter un check : `age IS NULL OR age >= 0`.
5. Supprimer la colonne `phone` (après vérification qu'elle est vide).

### Exercice 4 — Conventions (≈ 20 min)

Auditer la table suivante et proposer une **version conforme aux conventions** de la section 8 :

```sql
CREATE TABLE Customer (
    customerID INTEGER PRIMARY KEY,
    Customer_Name VARCHAR(100),
    AGE INTEGER,
    EmailAddr VARCHAR(255) UNIQUE,
    OrderCount INTEGER DEFAULT 0
);
```

Lister chaque non-conformité et la corriger.

### Exercice 5 — DROP prudent (≈ 15 min)

1. Créer une table `temp_data` avec 3 colonnes.
2. La supprimer **dans une transaction** : `BEGIN; DROP TABLE temp_data; SELECT 1; ROLLBACK;`.
3. Vérifier qu'elle existe encore.
4. La supprimer pour de bon.
5. La recréer puis la dropper avec `DROP TABLE IF EXISTS` pour démontrer l'idempotence.

## 10. Mini-défi de synthèse — base relationnelle complète (≈ 2 heures)

Concevoir et créer une **base de données relationnelle** pour un domaine au choix :

**Domaines suggérés** (au moins **4 tables** liées) :

1. **Bibliothèque** : `authors`, `books`, `members`, `loans`.
2. **Blog** : `users`, `posts`, `comments`, `tags`, `post_tags`.
3. **E-commerce** : `customers`, `products`, `orders`, `order_items`.
4. **Restaurant** : `categories`, `dishes`, `tables`, `reservations`.

### Exigences techniques

- [ ] Toutes les tables ont une **PK** (`id SERIAL` ou `UUID`).
- [ ] Toutes les FK sont déclarées avec une **stratégie `ON DELETE`** explicite et justifiée en commentaire.
- [ ] Au moins **un `UNIQUE`** sur une colonne métier (email, sku, isbn...).
- [ ] Au moins **deux `CHECK`** sur des contraintes métier (`price > 0`, `quantity >= 1`...).
- [ ] Toutes les colonnes sont **`NOT NULL`** sauf justification.
- [ ] Les **timestamps** `created_at` (et `updated_at` si pertinent) sont présents.
- [ ] Les noms suivent strictement les **conventions** de la section 8.

### Étapes

1. **Schéma sur papier** ou en ASCII art (diagramme ER simple).
2. **`CREATE TABLE`** pour chaque table.
3. **Inserts** : au moins 5 lignes par table, avec données cohérentes (respect des FK).
4. **3 requêtes de validation** :
   - Une `SELECT` qui montre une relation (avec un `WHERE`).
   - Une tentative d'insertion qui **doit** violer une FK ou un CHECK.
   - Un `ALTER TABLE` qui ajoute une colonne pertinente.

### Critères de validation

- [ ] Le script est **rejouable** (`DROP TABLE IF EXISTS` en tête, scripts dans le bon ordre des dépendances FK).
- [ ] Toutes les contraintes sont effectivement testées par au moins un cas d'erreur.
- [ ] Le schéma n'a aucune redondance évidente (cf. formes normales — M9).
